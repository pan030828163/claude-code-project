## 今日掌握

对照昨日"明日拓展"的目标——把 excel-mcp 的 mcp-excel.exe 服务端跑通——今日完成。GitHub Releases 下载了 v1.8.66 版本，解压到 tools/ExcelMcp/，修改 MCP 配置指向本地 exe，重启 Claude Code 后 Excel 工具全部可用。实测走通了完整流程：打开空工作簿 → 创建生鲜超市采购表（含 20 行模拟数据）→ 添加销量列 → 排序 → 两个子数据表 → 柱状图和饼图 → 保存 xlsx。Excel COM 接口的 session isolation 问题未触发——因为所有操作都在同一个 Claude Code 进程内串行完成，没有跨会话竞争。

但今日真正的进展不在 Excel，而在"管道拓宽"。先是 Playwright MCP 打通了浏览器自动化——从 B 站搜索 Claude Code 视频到打开扇贝英语单词表页面，验证了基于网页 DOM 快照的操控链路可行。随后扇贝单词表的数据爬取方案确立：浏览器快照提取单词列表，替代之前设想的截图 OCR 路线，数据更结构化、无损耗。这解决了昨天 CET6 学习工作流的第一个卡点——单词获取。

第三个管道是图像识别。最初想用阿里云 Qwen-VL 直接识别用户微信发来的艾宾浩斯表格截图，API 调通后确实完整提取了 19 个六级词，但用户敏锐指出了信息损耗问题——圈画、高亮、手写标注这些视觉标记全部蒸发。这暴露了多模型管线的一个本质局限：每次"翻译"（图文→文字→另一模型理解）都是一次信息熵增，精细的个人化痕迹最先丢失。

最后处理了 cc-connect daemon 宕机问题。原因是午后在另一个 Claude Code 会话中清理 RPA 相关文件时顺带停掉了 cc-connect 进程，而 scheduled task 只在登录时触发一次，之后无自动恢复机制。这本质上是一个进程生命周期管理漏洞——后台服务缺少 health check 和自动拉起。

## 相关知识库

### Playwright MCP 的 DOM 快照机制

Playwright MCP 不是给 Claude Code 一个"浏览器窗口"，而是每次命令后返回当前页面的可访问性树（Accessibility Tree）的快照文本。这个快照包含页面上每个可交互元素的 role、name、文本内容和唯一引用 ID（如 `ref=e123`）。Claude 看到的是纯文本版的页面结构而非像素渲染，所以可以"读懂"页面内容并精准定位按钮链接。但这种快照有两个盲区：一是纯视觉装饰元素（背景图、CSS 图标）不进入 Accessibility Tree，Claude 看不到；二是动态加载内容（瀑布流、懒加载图片）可能只在滚动后才出现在快照中。另外快照不是增量更新的——每次操作后返回完整快照，页面复杂的场景下 token 消耗非常大。

- Playwright Accessibility Snapshot: https://playwright.dev/docs/accessibility-testing
- MCP Playwright Server: https://github.com/microsoft/playwright-mcp

### 多模态管线中的信息衰减

当一条信息经过"图片→视觉模型→文字→语言模型→用户"这条链路时，每经过一个转换节点就会丢失一部分信息。丢失的不是"数据"而是"语境"——视觉模型能把表格里的单词和释义逐字提取出来，但它无法理解"这个词被红笔圈了三圈"意味着这是你的顽固生词。这就是为什么截图 OCR 路线在表格数据提取上能做到 95% 准确率，但在理解用户个人化学习痕迹上几乎是零。解决思路不是换更强的模型，而是把"个人化标记"从图片层提到数据层——在单词入库时让你直接告诉我哪些是重点，而不是等识别完之后再补。

- Qwen-VL 能力边界: https://help.aliyun.com/zh/model-studio/getting-started/models
- MCP 多工具编排: https://modelcontextprotocol.io/docs/concepts/architecture

### Windows Scheduled Task 的触发模型

Windows 任务计划程序支持多种触发器类型：LogonTrigger（登录时触发）、TimeTrigger（定时触发）、RegistrationTrigger（任务注册时触发）等。cc-connect daemon 配置的是 LogonTrigger，这意味着它在每次用户登录 Windows 时启动一次——启动后 daemon 进程持续运行。但如果进程被外部因素 kill（如其他程序误杀、用户手动终止），scheduled task 不会感知也不会自动重启，因为 Windows 认为"任务已成功触发并完成"。要实现进程死亡后自动恢复，有两种方案：一是改用重复间隔触发（如每 5 分钟检查一次），但会引入短暂的停机窗口；二是用真正的 Windows Service 而非 scheduled task，Service Control Manager 自带失败重启策略。cc-connect 选 schtasks 路线是出于跨平台兼容性（Linux systemd / macOS launchd / Windows schtasks 统一抽象），代价就是缺少原生的进程守护能力。

- Windows Task Scheduler Triggers: https://learn.microsoft.com/en-us/windows/win32/taskschd/task-triggers
