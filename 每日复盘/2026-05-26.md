## 相关知识库

### Windows COM 进程间通信

COM（Component Object Model）是 Windows 平台的二进制接口标准，允许一个应用程序（客户端）调用另一个应用程序（服务端）暴露的对象和方法。Excel MCP 的底层就是通过 COM 接口操作 Excel 应用程序实例：mcp-excel.exe 调用 `CoCreateInstance` 获取 Excel Application 对象，然后通过 IDispatch 接口动态调用其方法（Open、Range、Format 等）。COM 调用走的是同一台机器上的 RPC（Remote Procedure Call）通道，不经过网络协议栈，所以速度很快但受限于进程权限和会话隔离——如果 MCP 服务和 Excel 不在同一个用户会话，调用会失败。

- COM 技术概述: https://learn.microsoft.com/en-us/windows/win32/com/component-object-model--com--portal
- Excel COM 对象模型: https://learn.microsoft.com/en-us/office/vba/api/overview/excel/object-model

### MCP 服务器进程管理

MCP（Model Context Protocol）服务器是独立进程，与 Claude Code 之间通过 JSON-RPC 通信。传输层有两种模式：stdio（标准输入输出管道，Claude Code 启动时 spawn 子进程）和 HTTP（独立服务监听端口，Claude Code 作为客户端连接）。excel-mcp 用的是 command 模式——在配置文件中指定可执行文件路径，Claude Code 在需要时自动 spawn 该进程，通过 stdin/stdout 传递 JSON-RPC 请求和响应。这意味着不需要用户手动启动服务，Claude Code 负责进程生命周期管理。但如果进程崩溃，Claude Code 需要重新 spawn，这中间的状态恢复（如之前打开的工作簿）由 MCP 服务器自身负责。

- MCP 传输层规范: https://spec.modelcontextprotocol.io/specification/2024-11-05/basic/transports/
- Claude Code MCP 配置: https://docs.anthropic.com/en/docs/claude-code/mcp

### Git 凭证与网络代理分离

今天遇到的场景是：git push 走 SSH 端口 22，npm/npx 走 HTTPS 走 npmmirror 镜像，curl/wget 直连 GitHub Releases 走 443，WebFetch 工具另有自己的 TLS 链路。四个通道的出口不同、代理配置也不同。Git 可以通过 `http.proxy` 或 `core.gitProxy` 单独配代理，npm 有自己的 `registry` 和 `proxy` 配置项，系统级 `HTTP_PROXY` 环境变量对大部分 CLI 工具生效但不覆盖显式配置。理解这种"每条链路独立配代理"的架构，对国内开发环境下的网络问题排查至关重要。

- Git 代理配置: https://git-scm.com/docs/git-config#Documentation/git-config.txt-httpproxy
- npm 代理配置: https://docs.npmjs.com/cli/v9/using-npm/config#proxy

### 项目文件组织原则

今天确立了一个实践原则：工具二进制和运行时尽量放在项目目录而非 C 盘。好处是多方面的——C 盘空间不受挤压、项目可整体迁移到其他机器（相对路径）、git 管理下可追踪版本。excel-mcp 的服务端放在 `J:\Claude Code\Claude Code\tools\ExcelMcp\`，Skill 定义放在 `.claude/skills/excel-mcp/`，两者分离但都在项目根目录下。这种"工具二进制 + 配置文件 + Skill 定义"三元分离的布局，是 AI 编程工具链的可复用组织模式。

- 项目目录结构最佳实践: https://github.com/sbroenne/mcp-server-excel
